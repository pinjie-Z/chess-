# 揭棋规则设计审阅版

## 1. 文档目的

这份文档是给你审阅用的中文规则设计说明，对应英文正式版 `docs/RULE_DESIGN.md`。

它定义第一版 Spring Boot 服务端要实现哪些揭棋规则、哪些复杂规则暂缓。实现时服务端必须负责棋盘状态、合法走法判断、暗子揭示、超时、胜负和棋谱记录。浏览器客户端可以辅助显示，但不能决定规则结果。

## 2. 第一版实现范围

第一版要实现：

- 9 列 10 行棋盘。
- 全局坐标系统。
- 车、马、炮、兵、士、象、将/帅的基本走法。
- 暗子第一次移动时，按它所在原始位置的棋子类型走。
- 暗子第一次合法移动或吃子后，由服务端从剩余暗子池随机揭示真实类型。
- 吃子规则，包括“吃到暗子时，吃子方知道真实类型，被吃方不知道真实类型”。
- 两个将/帅不能无遮挡直接照面。
- 吃掉将/帅直接获胜。
- 认输直接结束。
- 服务端计时，超时直接结束。
- 40 回合无吃子判和。这里按程序实现口径记录为 80 个半回合，也就是双方合计走 80 步没有吃子。
- 本地棋谱记录，至少能支持调试和结果留存。

第一版暂不实现：

- 长将。
- 长捉。
- 兵长将 / 兵长捉的特殊判定。
- 求和、同意求和、拒绝求和流程。
- 完整的将死 / 困毙搜索。
- 多房间并发的复杂管理。

## 3. 关于“自杀送将”

第一版不做完整的“不能让自己被将军”判断。

也就是说，如果一方走了一步，虽然这步本身符合棋子走法，但走完后自己的将/帅暴露成下一步会被对方吃掉，这步第一版可以接受。之后对方直接吃掉将/帅即可获胜。

但是有一个例外必须判非法：如果走完后两个将/帅在同一列，中间没有任何棋子遮挡，也就是“双将照面”，这步不能接受。

这个设计是为了降低第一版规则复杂度。完整自我将军判断、将死判断、困毙判断都放到后续版本。

## 4. 坐标系统

WebSocket 协议和服务端统一使用全局坐标：

- 列从左到右是 `a` 到 `i`。
- 行是 `0` 到 `9`。
- `e0` 是红方帅初始位置。
- `e9` 是黑方将初始位置。
- 红方先手，在棋盘底部。
- 黑方在棋盘顶部。

内部二维数组建议这样映射：

```text
cells[row][col]
row = 9 - y
col = x - 'a'
y = 9 - row
x = (char) ('a' + col)
```

所以：

- 内部 `row = 0` 对应协议坐标第 `9` 行。
- 内部 `row = 9` 对应协议坐标第 `0` 行。

实现时不要在规则代码里到处传字符串坐标，应该先解析成 `Position` 对象。

坐标校验规则：

- `x` 必须是 `a` 到 `i`。
- `y` 必须是 `0` 到 `9`。
- 起点和终点不能相同，因为原地翻棋不允许。

## 5. 初始棋盘

初始位置按中国象棋标准摆法。

红方：

| 坐标 | 原始位置类型 |
|---|---|
| `a0`, `i0` | 车 |
| `b0`, `h0` | 马 |
| `c0`, `g0` | 象 |
| `d0`, `f0` | 士 |
| `e0` | 帅 |
| `b2`, `h2` | 炮 |
| `a3`, `c3`, `e3`, `g3`, `i3` | 兵 |

黑方：

| 坐标 | 原始位置类型 |
|---|---|
| `a9`, `i9` | 车 |
| `b9`, `h9` | 马 |
| `c9`, `g9` | 象 |
| `d9`, `f9` | 士 |
| `e9` | 将 |
| `b7`, `h7` | 炮 |
| `a6`, `c6`, `e6`, `g6`, `i6` | 卒 |

开局只有两个将/帅明置，其余 30 个非将/帅棋子都是暗子。

暗子有两个类型概念：

- `originalType`：原始位置类型，用于暗子第一次移动前的走法判断。
- `revealedType`：揭示后的真实类型，用于之后的走法判断。

暗子第一次走之前，用 `originalType` 判断。揭示之后，用 `revealedType` 判断。是否明子由 `revealedType` 是否存在推导，不单独保存布尔字段。

## 6. 领域对象建议

规则逻辑不应该堆在 WebSocket 处理器里。建议至少拆出这些对象：

| 对象 | 责任 |
|---|---|
| `Position` | 坐标值对象，负责解析、格式化、校验。 |
| `PieceColor` | 红/黑，以及对手颜色、前进方向。 |
| `PieceType` | 将/车/马/炮/兵/士/象。 |
| `Piece` | 棋子颜色、原始类型、揭示类型、是否可见、唯一 id。 |
| `Board` | 10x9 棋盘、查子、移动、路径阻挡辅助方法。 |
| `Move` | 起点、终点、走子方、服务端收到时间、客户端 `isFlip` 提示。 |
| `MoveValidationResult` | 合法/非法结果、错误码、提示、吃子/揭示信息。 |
| `GameState` | 棋盘、当前回合、游戏状态、计时、无吃子计数、步数。 |
| `FlipPool` | 每方剩余暗子真实类型池，由服务端随机抽取。 |
| `MoveRecord` | 已接受走法的记录，用于本地棋谱。 |
| `RuleEngine` | 根据 `GameState` 判断走法是否合法，不关心 WebSocket。 |

这样满足作业“至少 5 个领域类”的要求，也能避免控制器/处理器太臃肿。

## 7. 暗子池

每方都有一个暗子池，包含 15 个非将/帅真实类型：

```text
车 x2
马 x2
炮 x2
象 x2
士 x2
兵/卒 x5
```

将/帅不在暗子池里，因为开局已经明置。

暗子第一次成功移动或吃子时：

1. 服务端从该方剩余暗子池随机抽一个类型。
2. 该棋子变为明子。
3. 棋子的 `revealedType` 设置为抽出的类型。
4. 该类型从暗子池移除。
5. 服务端在 `moveResult.flipResult` 里发送揭示结果。

如果一个暗子还没走就被吃：

- 它从棋盘移除。
- 服务端仍然需要从被吃方暗子池确定它的真实类型。
- 该类型从被吃方暗子池移除。
- 吃子方收到真实 `capturedPiece`。
- 被吃方收到 `capturedPiece: "NULL"`。

如果被吃的是明子，双方都可以知道被吃类型。

## 8. 合法走法校验流程

建议按这个顺序校验：

1. 解析并校验起点、终点坐标。
2. 确认游戏状态是进行中。
3. 确认发送者属于这个房间。
4. 确认轮到发送者走。
5. 确认起点有棋子。
6. 确认起点棋子属于发送者。
7. 确认终点没有己方棋子。
8. 决定本步使用的移动类型：
   - 暗子：用 `originalType`。
   - 明子：用 `revealedType`。
9. 校验该棋子的走法形状和阻挡规则。
10. 模拟走子，若造成双将照面则拒绝。
11. 真正应用吃子、揭示、切换回合、重置计时、更新无吃子计数、检查胜负。

非法走法不能改变棋盘、回合、计时器、暗子池或棋谱。

## 9. 各棋子走法

### 9.1 将/帅

将/帅每次走一格，只能横向或纵向。

第一版将/帅限制在九宫内：

- 红方九宫：`d/e/f` 列，`0/1/2` 行。
- 黑方九宫：`d/e/f` 列，`7/8/9` 行。

任何接受后的局面都不能出现双将照面。

### 9.2 车

车横走或竖走任意格，中间不能有棋子。

### 9.3 马

马走“日”字：

- 横向 1、纵向 2，或横向 2、纵向 1。
- 蹩马腿位置必须为空。

如果纵向走两格，马腿是纵向相邻一格。如果横向走两格，马腿是横向相邻一格。

### 9.4 炮

炮横走或竖走。

不吃子时：

- 起点和终点必须同行或同列。
- 中间不能有棋子。
- 终点必须为空。

吃子时：

- 起点和终点必须同行或同列。
- 终点必须是敌方棋子。
- 中间必须正好隔一个棋子。

### 9.5 兵/卒

兵/卒每次走一格。

- 红方向 `y` 增大的方向前进。
- 黑方向 `y` 减小的方向前进。

过河前：

- 红方 `y <= 4` 算未过河。
- 黑方 `y >= 5` 算未过河。
- 只能向前走一格。

过河后：

- 可以向前走一格。
- 可以左右走一格。
- 永远不能后退。

### 9.6 士

士斜走一格。

揭棋强化规则：

- 暗士第一次移动时，按原始士走法，并受九宫限制。
- 士揭示后，仍然斜走一格，但可以离开九宫。

### 9.7 象

象斜走两格。

象眼位置必须为空。

揭棋强化规则：

- 暗象第一次移动时，按原始象走法，并受不能过河限制。
- 象揭示后，仍然斜走两格，但可以过河。

## 10. 应用一步合法走法

接受一步合法走法后：

1. 如果终点有敌方棋子，先移除被吃棋子。
2. 把起点棋子移动到终点。
3. 如果移动的棋子是暗子，从本方暗子池揭示。
4. 如果吃掉的是敌方暗子，从敌方暗子池确定被吃真实类型。
5. 更新无吃子计数：
   - 有吃子则清零。
   - 没吃子则加 1。
6. 写入 `MoveRecord`。
7. 检查是否结束。
8. 如果未结束，切换回合。
9. 重置下一名玩家的服务端计时。

客户端传来的 `isFlip` 只能当提示，不能当真实规则依据。是否揭示由服务端判断。

## 11. 胜负与和棋

第一版结束条件：

| 条件 | 结果 |
|---|---|
| 将/帅被吃 | 吃子方获胜。对外 `gameOver.reason` 使用 `checkmate`。 |
| 玩家发送 `Resign` | 对手获胜。对外 `gameOver.reason` 使用 `resign`。 |
| 玩家服务端计时超时 | 对手获胜。发送 `timeout` 消息。 |
| 连续 80 个半回合无吃子 | 内部状态和棋，棋谱记录和棋。对外协议若没有标准和棋字段，先作为项目扩展或记录字段处理。 |

第一版不做完整将死/困毙搜索。最小可玩闭环里，直接吃将/帅即可结束。

## 12. 超时

默认每步 60 秒。

服务端记录：

- `turnStartedAt`：本回合开始时间。
- `turnDeadlineAt`：本回合截止时间。
- 当前走子方颜色。

超时判断只能使用服务端时间。客户端时间戳只能用于展示，不能作为判定依据。

超时后：

- 游戏状态变为结束。
- 向双方广播 `timeout`。
- 该局之后不再接受任何走子。

## 13. 棋谱记录

棋谱可以先设计成 JSON Lines 或每局一个 JSON 文件。具体格式留到实现设计定。

每步成功走法至少记录：

- `roomId`。
- 步号。
- 走子颜色。
- 起点和终点。
- 本步用于校验的移动类型。
- 移动前是否是暗子。
- `flipResult`，如果本步揭示。
- 被吃棋子的服务端真实类型，如果有吃子。
- 如果双方收到的 `capturedPiece` 不同，也记录双方看到的值。
- 服务端时间戳。
- 本步后的无吃子计数。
- 如果本步后游戏结束，记录结果。

棋谱文件是服务端本地文件，可以包含完整隐藏信息；这些隐藏信息不能提前发给客户端。

## 14. 测试重点

实现规则时建议先写规则单元测试，再接 WebSocket：

- 坐标解析和数组映射。
- 初始棋盘摆放。
- 车的路径阻挡。
- 马的蹩马腿。
- 炮的隔子移动和隔子吃子。
- 红黑兵/卒过河前后移动。
- 士揭示前受九宫限制，揭示后不受九宫限制。
- 象揭示前不能过河，揭示后可以过河。
- 暗子第一次移动使用原始位置类型。
- 揭示后暗子池减少一个类型。
- 吃到暗子时双方看到的 `capturedPiece` 不同。
- 双将照面被拒绝。
- 吃将/帅结束游戏。
- 无吃子计数的增加和清零。

## 15. 需要你重点确认的决定

请重点看这几个决定是否接受：

1. 第一版不做长将、长捉、求和、完整将死/困毙搜索。
2. 普通“自杀送将”不拦截，双将照面仍拦截。
3. 暗子没走就被吃时，由服务端从被吃方暗子池确定真实类型。
4. 士/象揭示前受原始限制，揭示后按揭棋强化规则可离宫/过河。
5. 40 回合无吃子按 80 个半回合实现。
